**Bead:** `agents-config-w1qls.1.2`

## Summary

A.2 delivers `packages/installer/src/installer/core/model.py` — the pure data layer every later engine module reads. Enums, dataclasses, one type alias. No behaviour beyond construction, equality, and frozen-immutability where called for. The umbrella spec assigns this module the role of "in-memory shape of the staging plan and its supporting types"; A.2 establishes that shape so B.1+ can compose against it without churn.

## 1. Scope & Exit Criteria

### In scope

- `core/__init__.py` (empty package marker)
- `core/model.py` — the types enumerated below
- `tests/unit/test_model.py` — construction, equality, immutability, enum coverage

### Explicitly NOT in scope

- `Config` and `ToolPaths` — the bead acceptance criteria mention them, but the umbrella spec parks them in `config.py` (story B.1). A.2 does **not** pre-stub them; B.1 will introduce them where they belong.
- `Plugin` enum — deliberately omitted (see §3.2 below). Plugins will be string-keyed via a registry built in F.1.
- Behaviour: no DYNAMIC-INCLUDE flattening, no merge logic, no IO. `IncludeDirective` variants are markers; their semantics live in `core/templates.py` (Epic C).
- `MergeStrategy` protocol — Epic E.
- Per-tool adapters — Epic B+.

### Exit criteria

1. `uv run pytest packages/installer/tests/unit/test_model.py -q` passes.
2. `uv run ruff check packages/installer && uv run mypy --strict packages/installer/src` clean on the new module and tests.
3. CI green on the PR.

## 2. Divergence from the umbrella spec

The umbrella spec calls for "`Tool` and `Plugin` enums for exhaustive type-checking; the corresponding `ToolAdapter` / `PluginAdapter` instances live in registries keyed by enum value." A.2 deliberately diverges on the `Plugin` half:

- **`Tool` stays as an enum.** Each tool has hand-written adapter behaviour (OpenCode XDG path, OpenCode's "skip shared `agents/`" rule, Gemini frontmatter transform, Codex placeholder extensions). Adding a tool already requires code; the enum gives mypy exhaustive matching in return.
- **`Plugin` is dropped.** Plugins are overlays — discoverable by scanning `src/plugins/<name>/`. A string-keyed registry supports zero-code addition of new plugins. The umbrella spec will be amended in a follow-up bead to reflect this.

## 3. Module contents

All types live in `core/model.py`. Imports kept stdlib-only — this module is the foundation, not a consumer.

### 3.1 `Tool` enum

```python
class Tool(StrEnum):
    CLAUDE = "claude"
    CODEX = "codex"
    GEMINI = "gemini"
    OPENCODE = "opencode"
```

`StrEnum` so `Tool("claude") == Tool.CLAUDE` and so values render naturally in logs and provenance. Member set is closed; expansion requires code (intentional — see §2).

### 3.2 No `Plugin` enum

Plugins are string-keyed throughout. Their registry (F.1) maps `str → PluginAdapter`. Discovery walks `src/plugins/<name>/` and registers each directory. Provenance carries the plugin's name string — see §3.4.

### 3.3 `FileKind` enum

```python
class FileKind(StrEnum):
    DIR = "dir"
    SETTINGS_JSON = "settings.json"
    JSONC = "jsonc"
    TOML = "toml"
    NAMESPACED_MD = "namespaced_md"
    OTHER = "other"
```

Mirrors bash `classify_file()` (`scripts/install.sh:486-505`). The bash function returns dynamic strings like `"rules.md"` / `"commands.md"`; the Python model splits that into `kind=NAMESPACED_MD` + a separate `namespace: str | None` field on `StagedItem`. Merge dispatch keys on `(kind, namespace)`, so duplicating namespace info into the enum buys nothing and costs exhaustiveness.

### 3.4 `Provenance`

```python
@dataclass(frozen=True, slots=True)
class Provenance:
    kind: Literal["tool", "plugin"]
    name: str
```

Tagged origin tracking on every `StagedItem`. The `kind` discriminator prevents collision in the (currently hypothetical) case where a plugin shares a tool's name. `Literal` enforces the discriminator at type-check time; runtime construction with an invalid string is permitted (a known Python limitation we accept).

### 3.5 `IncludeDirective` — discriminated union

```python
@dataclass(frozen=True, slots=True)
class FileInclude:
    path: Path  # repo-relative

@dataclass(frozen=True, slots=True)
class AllRulesInclude:
    pass  # marker; semantics in core/templates.py

IncludeDirective: TypeAlias = FileInclude | AllRulesInclude
```

Two variants, modelled separately for `match`-exhaustiveness under mypy `--strict`:

- `FileInclude` — verbatim substitution of one file's bytes.
- `AllRulesInclude` — glob rules from the staged tree, sort lexicographically, join with `\n---\n`.

Implementation in `core/templates.py` may reuse the same file-read primitive for both variants (an `AllRulesInclude` is effectively many `FileInclude` reads + a join). The distinct dataclass per variant exists at the type level, not as a claim about implementation duplication.

A `NamedRulesInclude` variant from the umbrella spec is intentionally omitted — no current use case.

### 3.6 `StagedItem`

```python
@dataclass(frozen=True, slots=True)
class StagedItem:
    source_path: Path        # absolute source location
    dest_relpath: Path       # path relative to tool's dest root
    kind: FileKind
    namespace: str | None    # parent-dir label for NAMESPACED_MD; None otherwise
    provenance: Provenance
    content: bytes           # eager — read at staging time
```

Frozen so staging emits immutable items and merge strategies produce new `StagedItem`s rather than mutating. `content: bytes` is **eager** by design: the umbrella spec's sync phase hashes every staged file for the skip-compare, so a lazy `Path`-only model wouldn't save reads — it would only spread the read across two phases. Eager content also gives strategies a uniform `(existing, incoming) -> StagedItem` shape without helper methods.

### 3.7 `StagingPlan`

```python
@dataclass(slots=True)
class StagingPlan:
    items: dict[Path, StagedItem]  # keyed by dest_relpath
    tool: Tool
```

**Not frozen** — staging builds the plan incrementally. Keyed by `dest_relpath` so that collisions (two sources targeting the same dest path) are surfaced as `dict` overwrites and resolved by the merge engine. `tool` ties the plan to its owning tool for downstream sync.

### 3.8 `Orphan`

```python
@dataclass(frozen=True, slots=True)
class Orphan:
    tool: str             # bucket label: "claude", "codex", "gemini", "opencode", "beads"
    namespace: str        # "commands" | "skills" | "agents" | "rules" | "formulas"
    path: Path            # absolute path of the orphan
    kind: Literal["dir", "file"]
```

Replaces the four parallel arrays at `scripts/install.sh:1456-1467` (`ORPHAN_TOOLS` / `ORPHAN_NS` / `ORPHAN_PATHS` / `ORPHAN_KINDS`) — one record per orphan rather than four index-aligned arrays. `tool` is `str` (not `Tool`) because the orphan bucket includes plugin namespaces (`"beads"`) that aren't tools.

### 3.9 `Counters`

```python
@dataclass(slots=True)
class Counters:
    staged: int = 0
    created: int = 0
    updated: int = 0
    skipped: int = 0
    pruned: int = 0
    backed_up: int = 0
```

Run-level totals reported at install completion. **Not frozen** — incremented during the run.

## 4. Invariants

- Frozen dataclasses (`Provenance`, `FileInclude`, `AllRulesInclude`, `StagedItem`, `Orphan`) gain dataclass-generated `__eq__` and `__hash__`. All field types are hashable (`bytes`, `Path`, `str`, `int`, enums), so the items themselves are hashable.
- Mutable dataclasses (`StagingPlan`, `Counters`) intentionally omit `__hash__` — instances are compared by inspecting their fields.
- Enum members are unique by value; iteration order is declaration order.
- `Literal` discriminators (`Provenance.kind`, `Orphan.kind`) are enforced at type-check time; runtime values outside the literal set are not raised against (Python limitation; documented, not worked around).

## 5. Test plan

Single test module: `packages/installer/tests/unit/test_model.py`. All tests are pure construction/equality/immutability checks — no IO, no fixtures.

Test names express the contract, not the implementation.

### Enums
1. `test_tool_enum_has_all_four_members` — set of `Tool` equals `{CLAUDE, CODEX, GEMINI, OPENCODE}`; values match the lowercase names.
2. `test_tool_string_lookup_round_trip` — `Tool("claude") is Tool.CLAUDE`.
3. `test_file_kind_enum_has_expected_members` — set equals `{DIR, SETTINGS_JSON, JSONC, TOML, NAMESPACED_MD, OTHER}`.

### Frozen dataclasses
4. `test_provenance_equality` — same `(kind, name)` compare equal; differing kind or name do not.
5. `test_provenance_is_frozen` — assigning to `name` raises `FrozenInstanceError`.
6. `test_file_include_construction_and_equality` — `FileInclude(Path("a"))` equals another with same path; differs from `FileInclude(Path("b"))`.
7. `test_all_rules_include_is_singleton_by_value` — two `AllRulesInclude()` instances compare equal.
8. `test_staged_item_equality_and_frozen` — full-field equality; frozen.
9. `test_orphan_equality_and_frozen` — full-field equality; frozen.

### Discriminated union
10. `test_include_directive_match_dispatches_to_both_arms` — runtime `match` over `FileInclude(...)` and `AllRulesInclude()` reaches each arm exactly once. Static exhaustiveness (mypy catching a missed variant after a future addition) is enforced separately via `typing.assert_never` in the production code paths that consume `IncludeDirective`, not in this unit test.

### Mutable dataclasses
11. `test_staging_plan_starts_empty_and_accepts_items` — construct with empty dict, add a `StagedItem`, read it back by key.
12. `test_staging_plan_keyed_by_dest_relpath` — overwriting the same key replaces the prior entry (collision surface check).
13. `test_counters_default_to_zero` — fresh `Counters()` has every field at 0.
14. `test_counters_increment_isolated` — incrementing one field leaves the others untouched.

### Hashability
15. `test_frozen_items_are_hashable` — `StagedItem` and `Orphan` can be placed in a `set`.

Total: 15 unit tests. Expected runtime: well under one second. No integration or golden-master tests — this story has no behaviour.

## 6. Forward-compatibility notes

- When B.1 introduces `Config` / `ToolPaths`, those land in `config.py` and may import from `model.py` freely (model has no dependencies on config).
- When F.1 introduces plugin discovery, the plugin registry's keys are plain strings; `Provenance(kind="plugin", name="<discovered-name>")` is how staged items from plugins identify themselves.
- A.3 (`core/io_port.py`) does not import from `model.py` — they're sibling foundation modules, parallel-buildable.

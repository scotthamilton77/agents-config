**Bead:** `agents-config-w1qls.1.2`

## Summary

A.2 delivers `packages/installer/src/installer/core/model.py` — the pure data layer every later engine module reads. Enums, dataclasses, one type alias. No behaviour beyond construction, equality, and frozen-immutability where called for. The umbrella spec assigns this module the role of "in-memory shape of the staging plan and its supporting types"; A.2 establishes that shape so B.1+ can compose against it without churn.

## 1. Scope & Exit Criteria

### In scope

- `core/__init__.py` (package marker — single-line docstring, no exports)
- `core/model.py` — the types enumerated below
- `tests/unit/test_model.py` — construction, equality, immutability, discriminated-union dispatch, hashability (enum-membership tests deliberately omitted — see §5.1)

### Explicitly NOT in scope

- `Config` and `ToolPaths` — the bead acceptance criteria mention them, but the umbrella spec parks them in `config.py` (story B.1). A.2 does **not** pre-stub them; B.1 will introduce them where they belong.
- `Plugin` enum — deliberately omitted (see §3.2 below). Plugins will be string-keyed via a registry built in F.1.
- Behaviour: no DYNAMIC-INCLUDE flattening, no merge logic, no IO. `IncludeDirective` variants are markers; their semantics live in `core/templates.py` (Epic C).
- `MergeStrategy` protocol — Epic E.
- Per-tool adapters — Epic B+.

### Exit criteria

The installer package has its own `pyproject.toml` at `packages/installer/`; the project's `Makefile` `cd`s into that directory before invoking `uv`. Reproduce the gate the same way:

1. From `packages/installer/`: `uv run pytest tests/unit/test_model.py -q` passes.
2. From `packages/installer/`: `uv run ruff check src tests && uv run mypy --strict src` clean.
3. CI green on the PR.

## 2. Divergence from the umbrella spec

The umbrella spec calls for "`Tool` and `Plugin` enums for exhaustive type-checking; the corresponding `ToolAdapter` / `PluginAdapter` instances live in registries keyed by enum value." A.2 deliberately diverges on the `Plugin` half:

- **`Tool` stays as an enum.** Each tool has hand-written adapter behaviour (OpenCode XDG path, OpenCode's "skip shared `agents/`" rule, Gemini frontmatter transform, Codex placeholder extensions). Adding a tool already requires code; the enum gives mypy exhaustive matching in return.
- **`Plugin` is dropped.** Plugins are overlays — discoverable by scanning `src/plugins/<name>/`. A string-keyed registry supports zero-code addition of new plugins. The umbrella spec is amended **inline in this PR** to reflect the divergence (see §7); no follow-up bead is filed.

## 3. Module contents

All types live in `core/model.py`. Imports kept stdlib-only — this module is the foundation, not a consumer.

### 3.1 `Tool` enum

```python
class Tool(StrEnum):
    CLAUDE = "claude"
    CODEX = "codex"
    GEMINI = "gemini"
    OPENCODE = "opencode"
```

`StrEnum` so `Tool("claude") == Tool.CLAUDE` and so values render naturally in logs and provenance. Member set is closed; expansion requires code (intentional — see §2).

### 3.2 No `Plugin` enum

Plugins are string-keyed throughout. Their registry (F.1) maps `str → PluginAdapter`. Discovery walks `src/plugins/<name>/` and registers each directory. Provenance carries the plugin's name string — see §3.4.

### 3.3 `FileKind` enum

```python
class FileKind(StrEnum):
    DIR = "dir"
    SETTINGS_JSON = "settings.json"
    JSONC = "jsonc"
    TOML = "toml"
    NAMESPACED_MD = "namespaced_md"
    OTHER = "other"
```

Mirrors bash `classify_file()` (`scripts/install.sh:486-505`). The bash function returns dynamic strings like `"rules.md"` / `"commands.md"`; the Python model splits that into `kind=NAMESPACED_MD` + a separate `namespace: str | None` field on `StagedItem`. Merge dispatch keys on `(kind, namespace)`, so duplicating namespace info into the enum buys nothing and costs exhaustiveness.

### 3.4 `Provenance`

```python
@dataclass(frozen=True, slots=True)
class Provenance:
    kind: Literal["tool", "plugin"]
    name: str
```

Tagged origin tracking on every `StagedItem`. The `kind` discriminator prevents collision in the (currently hypothetical) case where a plugin shares a tool's name. `Literal` enforces the discriminator at type-check time; runtime construction with an invalid string is permitted (a known Python limitation we accept).

### 3.5 `IncludeDirective` — discriminated union

```python
@dataclass(frozen=True, slots=True)
class FileInclude:
    path: Path  # repo-relative

@dataclass(frozen=True, slots=True)
class AllRulesInclude:
    pass  # marker; semantics in core/templates.py

IncludeDirective: TypeAlias = FileInclude | AllRulesInclude
```

Two variants, modelled separately for `match`-exhaustiveness under mypy `--strict`:

- `FileInclude` — verbatim substitution of one file's bytes.
- `AllRulesInclude` — glob rules from the staged tree, sort lexicographically, join with `\n---\n`.

Implementation in `core/templates.py` may reuse the same file-read primitive for both variants (an `AllRulesInclude` is effectively many `FileInclude` reads + a join). The distinct dataclass per variant exists at the type level, not as a claim about implementation duplication.

A `NamedRulesInclude` variant from the umbrella spec is intentionally omitted — no current use case.

### 3.6 `StagedItem`

```python
@dataclass(frozen=True, slots=True)
class StagedItem:
    source_path: Path                # absolute source location
    dest_relpath: Path               # path relative to tool's dest root
    kind: FileKind
    namespace: str | None            # parent-dir label for NAMESPACED_MD; None otherwise
    provenance: Provenance
    content: bytes | None = None     # eager file bytes; None for FileKind.DIR (the sync engine recurses from source_path)
    executable: bool = False         # sync-phase mode bit (0o755 vs 0o644); ignored for DIR
```

Frozen so staging emits immutable items and merge strategies produce new `StagedItem`s rather than mutating. `content` is **eager** by design when present: the umbrella spec's sync phase hashes every staged file for the skip-compare, so a lazy `Path`-only model wouldn't save reads — it would only spread the read across two phases. Eager content also gives strategies a uniform `(existing, incoming) -> StagedItem` shape without helper methods. `content` is `bytes | None` (not bare `bytes`) only so `FileKind.DIR` items can stage as a single record without inventing placeholder bytes; for every non-DIR kind, callers MUST populate it.

`executable` is on `StagedItem`, not on `FileKind`, because the executable bit is a **sync-phase write attribute** (the mode argument to `os.open` / `Path.chmod`), not a **classification for merge dispatch**. A bash script and a markdown file may both need the same merge strategy (`last_wins_silent`) but different write modes. The current adapters only set `executable=True` for plugin scripts (e.g. beads' `~/.beads/scripts/*.sh`); future adapters that install Python or shell tooling under tools can opt in without a `FileKind` change. If a merge strategy ever needs to dispatch on "is this code" specifically, a `CODE` `FileKind` variant can be added later — that decision is orthogonal to the executable bit.

### 3.7 `StagingPlan`

```python
@dataclass(slots=True)
class StagingPlan:
    items: dict[Path, StagedItem]  # keyed by dest_relpath
    tool: Tool
```

**Not frozen** — staging builds the plan incrementally. Keyed by `dest_relpath`. A plain `dict` **silently overwrites** on duplicate keys — it does not by itself surface a collision to the merge engine. The collision-detection contract therefore lives in `core/staging.py`: callers MUST check `dest_relpath in items` BEFORE assignment and route through the merge registry on hit. `StagingPlan` is bare storage; the data layer deliberately holds engine behaviour out. `tool` ties the plan to its owning tool for downstream sync.

### 3.8 `Orphan`

```python
@dataclass(frozen=True, slots=True)
class Orphan:
    tool: str             # bucket label: "claude", "codex", "gemini", "opencode", "beads"
    namespace: str        # "commands" | "skills" | "agents" | "rules" | "formulas"
    path: Path            # absolute path of the orphan
    kind: Literal["dir", "file"]
```

Replaces the four parallel arrays at `scripts/install.sh:1456-1467` (`ORPHAN_TOOLS` / `ORPHAN_NS` / `ORPHAN_PATHS` / `ORPHAN_KINDS`) — one record per orphan rather than four index-aligned arrays. `tool` is `str` (not `Tool`) because the orphan bucket includes plugin namespaces (`"beads"`) that aren't tools.

### 3.9 `Counters`

```python
@dataclass(slots=True)
class Counters:
    staged: int = 0
    created: int = 0
    updated: int = 0
    skipped: int = 0
    pruned: int = 0
    backed_up: int = 0
```

Run-level totals reported at install completion. **Not frozen** — incremented during the run.

## 4. Invariants

- Frozen dataclasses (`Provenance`, `FileInclude`, `AllRulesInclude`, `StagedItem`, `Orphan`) gain dataclass-generated `__eq__` and `__hash__`. All field types are hashable (`bytes`, `Path`, `str`, `int`, enums), so the items themselves are hashable.
- Mutable dataclasses (`StagingPlan`, `Counters`) intentionally omit `__hash__` — instances are compared by inspecting their fields.
- Enum members are unique by value; iteration order is declaration order.
- `Literal` discriminators (`Provenance.kind`, `Orphan.kind`) are enforced at type-check time; runtime values outside the literal set are not raised against (Python limitation; documented, not worked around).

## 5. Test plan

Single test module: `packages/installer/tests/unit/test_model.py`. Every test pins **a design decision we made**, not Python language semantics. Tests that would only verify "Python enums work" or "dicts work as dicts" are deliberately omitted — see §5.1.

Test names express the contract, not the implementation.

### Frozen dataclasses — pin the `frozen=True` design choice
1. `test_provenance_equality` — same `(kind, name)` compare equal; differing kind or name do not. (Pins: equality-by-value as part of the type's contract; would catch a future `@dataclass(eq=False)` regression.)
2. `test_provenance_is_frozen` — assigning to `name` raises `FrozenInstanceError`. (Pins: immutability as part of the contract; would catch a future `frozen=False` regression.)
3. `test_file_include_construction_and_equality` — `FileInclude(Path("a"))` equals another with same path; differs from `FileInclude(Path("b"))`.
4. `test_file_include_is_frozen` — `FileInclude(...)` raises `FrozenInstanceError` on attribute assignment. (Pins: immutability of the variant.)
5. `test_all_rules_include_is_singleton_by_value` — two `AllRulesInclude()` instances compare equal (marker-class contract: identity-free equality).
6. `test_all_rules_include_is_frozen` — asserts `AllRulesInclude.__dataclass_params__.frozen is True`. (See the test docstring for why the standard `pytest.raises(FrozenInstanceError)` form does not work for empty `frozen=True, slots=True` dataclasses on current CPython — a known interaction between slot-class reparenting and the generated `__setattr__`'s `super()` call.)
7. `test_staged_item_equality_and_frozen` — full-field equality; frozen.
8. `test_staged_item_executable_defaults_false_and_round_trips` — fresh `StagedItem` has `executable is False`; an explicit `executable=True` override surfaces and breaks equality with the default. (Pins: the chosen default and the executable field's participation in equality.)
9. `test_staged_item_accepts_dir_kind_with_no_content` — `StagedItem(kind=FileKind.DIR, ...)` constructs with `content` omitted; `content is None` post-construction. (Pins: the `content: bytes | None = None` design that lets directories stage as a single record without inventing placeholder bytes.)
10. `test_orphan_equality_and_frozen` — full-field equality; frozen.

### Discriminated union — pin the variant set
11. `test_include_directive_match_dispatches_to_both_arms` — runtime `match` over `FileInclude(...)` and `AllRulesInclude()` reaches each arm exactly once. Pins the **two-variant contract** so that adding a third variant without updating its consumers raises mypy `assert_never` flags at the call sites (not in this test). The test itself just confirms the union admits both arms today.

### Mutable dataclasses — pin our chosen defaults
12. `test_counters_default_to_zero` — fresh `Counters()` has every field at 0. (Pins: the choice of `0` as the starting value for every counter; would catch a typo'd default. Counter increment behaviour is **not** tested — that's Python's `int` arithmetic.)

### Hashability — pin the set-membership contract
13. `test_frozen_items_are_hashable` — `StagedItem`, `Orphan`, `Provenance`, `FileInclude`, and `AllRulesInclude` placed in a `set` round-trip without `TypeError`. (Pins: all field types remain hashable; would catch a future field-type change that breaks set membership.)

Total: 13 unit tests. Expected runtime: well under one second. No integration or golden-master tests — this story has no behaviour.

### 5.1 Tests deliberately not written

Each of the following would test Python language machinery rather than our design:

- "`Tool` enum has four members" / "`FileKind` enum has six members" — set-membership of an enum is **variable structure**; a meaningful change to the enum should be caught by `mypy --strict` exhaustiveness at the consumer sites, not by a brittle membership snapshot test.
- "`Tool('claude') is Tool.CLAUDE`" — `StrEnum`'s value-lookup is a Python guarantee, not something we implemented.
- `StagingPlan` construction / dict get-set / collision-overwrite — `StagingPlan` is a thin record wrapping `dict[Path, StagedItem]` with no methods of its own; testing dict semantics on the field tests CPython.
- `Counters` increment isolation — `int +=` operates on independent fields by language guarantee; we add nothing.

If `StagingPlan` later grows methods (an `add()` helper, collision-detection callbacks, an immutability handoff for sync), tests for those land then.

## 6. Forward-compatibility notes

- When B.1 introduces `Config` / `ToolPaths`, those land in `config.py` and may import from `model.py` freely (model has no dependencies on config).
- When F.1 introduces plugin discovery, the plugin registry's keys are plain strings; `Provenance(kind="plugin", name="<discovered-name>")` is how staged items from plugins identify themselves.
- A.3 (`core/io_port.py`) does not import from `model.py` — they're sibling foundation modules, parallel-buildable.

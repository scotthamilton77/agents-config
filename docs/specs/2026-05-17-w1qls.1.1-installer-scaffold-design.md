# A.1 — uv-managed packages/installer/ Scaffold + CI Green — Design

**Bead:** `agents-config-w1qls.1.1`
**Parent epic:** `agents-config-w1qls.1` (Epic A — Foundation)
**Parent feature:** `agents-config-w1qls` (Python installer rewrite)

**Date:** 2026-05-17

## Summary

A.1 stands up `packages/installer/` as a uv-managed Python package and establishes the monorepo's CI dispatch precedent (top-level `Makefile` + single GitHub Actions workflow). It delivers a thin, working CLI surface (`scripts/install.py --help` exits 0) backed by a real smoke test and the complete lint / typecheck / coverage / audit / actionlint gate. After A.1 merges, A.2 and A.3 land into a package that already type-checks, lints, runs tests, measures coverage, audits deps, and reports CI green on every PR.

This is the first Python subproject in what is becoming a modular monorepo, so A.1's choices are precedent-setting for everything that follows.

## 1. Scope & Exit Criteria

### In scope

- `packages/installer/` directory tree (minimal — only what A.1 needs)
- `pyproject.toml` with full runtime + dev deps and all static-analysis config
- `uv.lock` (generated and committed)
- `packages/installer/README.md` (minimal placeholder)
- `src/installer/__init__.py` (empty) and `src/installer/cli.py` (argparse-based `main()` with `--help`)
- `tests/unit/test_cli_smoke.py` (3 unit tests)
- `scripts/install.py` (4-line entry stub)
- Top-level `Makefile` (monorepo dispatch — `ci` and per-package targets)
- `.github/workflows/ci.yml` (single workflow that runs `make ci`)
- `.gitignore` additions (coverage and cache artefacts)

### Explicitly NOT in scope (deliberate non-decisions)

- Installer behaviour of any kind (no source-walk, no staging, no sync, no merge engine)
- Empty placeholder modules under `core/`, `tools/`, `plugins/` — Total TDD says no production code without a failing test that justifies it
- `installer.toml` (Epic G concern)
- Integration suite, golden-master suite, fixture scaffolding (Epic B+ concerns)
- Pre-commit hooks (CI is the enforcement gate)
- Bandit (ruff `S` covers it), pyright (mypy strict + extra knobs is enough), vulture (ruff covers most), `interrogate` (no docstring-coverage requirement)
- Shellcheck for `scripts/install.sh` (install.sh stays as parity reference until Epic H)
- Any modification to `scripts/install.sh` itself

### Exit criteria (all must be green)

1. `make ci` from repo root exits 0 locally
2. `.github/workflows/ci.yml` reports green on the PR
3. `python3 scripts/install.py --help` prints usage and exits 0
4. `uv run pytest packages/installer/tests -q` — green (3/3 tests)
5. `uv run ruff check packages/installer` — clean
6. `uv run ruff format --check packages/installer` — clean
7. `uv run mypy --strict packages/installer/src` — clean
8. Coverage report shows ≥90% branch coverage on `packages/installer/src/installer/cli.py`
9. `uv run pip-audit` against the synced venv — clean (no known CVEs)
10. `uv run actionlint` against `.github/workflows/*.yml` — clean
11. `uv.lock` committed
12. Completion gate (quality-reviewer, simplify, verify-checklist) passes

### 2.2 Top-level `Makefile` — monorepo dispatch

```make
.PHONY: ci ci-installer test-installer lint-installer format-check-installer \
        typecheck-installer cov-installer audit-installer lint-actions \
        verify-entry-installer

INSTALLER := packages/installer

ci: ci-installer lint-actions

ci-installer: lint-installer format-check-installer typecheck-installer \
              test-installer cov-installer audit-installer \
              verify-entry-installer

test-installer:
	cd $(INSTALLER) && uv run pytest -q

lint-installer:
	cd $(INSTALLER) && uv run ruff check

format-check-installer:
	cd $(INSTALLER) && uv run ruff format --check

typecheck-installer:
	cd $(INSTALLER) && uv run mypy --strict src

cov-installer:
	cd $(INSTALLER) && uv run pytest --cov --cov-report=term-missing

audit-installer:
	cd $(INSTALLER) && uv sync --frozen && uv run pip-audit

lint-actions:
	cd $(INSTALLER) && uv run actionlint

verify-entry-installer:
	uv --project $(INSTALLER) run python scripts/install.py --help > /dev/null
```

### 2.5 `src/installer/cli.py` — minimal real surface

```python
import argparse


def main(argv: list[str] | None = None) -> int:
    parser = argparse.ArgumentParser(
        prog="installer",
        description="Install agent configurations for AI coding assistants.",
    )
    parser.parse_args(argv)
    return 0
```

The argparse-with-no-positionals-and-no-subcommands shape exists solely so `--help` works and exits 0. A.2/A.3 grow it with real subcommands; A.1 establishes only the entry point + `int` return contract.

## 3. Test Plan (load-bearing artifact)

Per parent spec §Total TDD: this is the artifact the implementation must satisfy, and it survives as a regression checklist.

### Tests file — `packages/installer/tests/unit/test_cli_smoke.py`

| # | Test name | Contract | Fixture |
|---|-----------|----------|---------|
| 1 | `test_main_returns_zero_with_no_args` | `main([])` returns `0`. Contract: no-arg invocation is a clean exit. Justifies `cli.main()` existing and returning an `int`. | none |
| 2 | `test_main_help_exits_with_status_zero` | `main(["--help"])` raises `SystemExit` with `code == 0`. Documents argparse's `--help` behaviour as A.1's contract (the AC literally requires this). | `pytest.raises(SystemExit)` |
| 3 | `test_main_help_prints_usage` | `main(["--help"])` writes captured stdout containing both `"usage:"` and `"installer"` (the prog name). Contract: help output is informative and identifies the tool. | `capsys` |

### Test shape (illustrative — implementer writes the literal final code)

```python
import pytest
from installer.cli import main


def test_main_returns_zero_with_no_args() -> None:
    assert main([]) == 0


def test_main_help_exits_with_status_zero() -> None:
    with pytest.raises(SystemExit) as exc_info:
        main(["--help"])
    assert exc_info.value.code == 0


def test_main_help_prints_usage(capsys: pytest.CaptureFixture[str]) -> None:
    with pytest.raises(SystemExit):
        main(["--help"])
    captured = capsys.readouterr()
    assert "usage:" in captured.out
    assert "installer" in captured.out
```

### Coverage analysis (against the 90% branch gate)

`cli.py` has two reachable branches: the normal path (return 0) and the `--help` SystemExit. Tests 1 and 2 hit both. Branch coverage is 2/2 = 100%. Clears the gate with margin.

### Entry-binding verification (NOT a pytest test)

The AC says "scripts/install.py prints --help and exits 0" — the *binding* between `scripts/install.py` and `installer.cli.main`. Subprocess-in-pytest tests are slow and brittle, so this lives in the Makefile (`verify-entry-installer`):

```make
verify-entry-installer:
	uv --project $(INSTALLER) run python scripts/install.py --help > /dev/null
```

Added to `ci-installer`'s dependency list. Fast (~1s), exercises the real entry path, no subprocess fixture noise in the unit suite.

## 5. Open Considerations & Risks

### Open question (raise now, decide during implementation)

The Makefile's `verify-entry-installer` target uses `uv --project packages/installer run python scripts/install.py --help`. This assumes `uv` will set up the venv on first run. If the dev's machine doesn't have Python 3.11 yet, `uv` auto-installs it (modern uv behaviour). CI uses `astral-sh/setup-uv@v3` + explicit `uv python install 3.11`, so it's deterministic there. Local devs may see a one-time slow first run. Document in `packages/installer/README.md`.

### Risks worth naming

1. **`actionlint-py` PyPI freshness** — the package bundles a Go binary; releases lag the upstream Go tool by a few weeks. Likely fine, but if it ever bit-rots, fallback is `brew install actionlint` + a conditional Makefile target. Low probability, easy mitigation.
2. **`pip-audit` against `uv.lock`** — `pip-audit` natively reads requirements.txt-style files, not `uv.lock`. The Makefile target uses `uv sync --frozen` then audits the synced venv, which works. Slight extra runtime (~5s for the sync), and any drift between `uv.lock` and what's installed gets surfaced as an audit failure rather than a sync failure. Acceptable.
3. **Coverage gate on argparse boilerplate** — argparse's internal branches (e.g., `--help` triggering an internal `SystemExit`) are partly in the stdlib and partly in our `main()`. With our test plan, branch coverage on `cli.py` itself is 100%. If the gate ever flickers below 90%, it'll be because someone added a real branch without testing it — exactly when we want the gate to bite. Working as intended.
4. **The PR will be larger than A.1's tiny test count suggests** — 12 new files, ~150 lines of configuration. Reviewer may push back asking "why so much for 3 tests?" The answer is in this design: the configs are the monorepo precedent, not A.1-specific noise. Flag this in the PR description.

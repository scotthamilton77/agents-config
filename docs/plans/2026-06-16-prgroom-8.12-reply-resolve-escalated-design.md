# Design: prgroom deterministic verbs — `_reply` (+ CONTEXTUAL memory routing) + `resolve-escalated`

> **Bead:** agents-config-abn9.8.12 · **Parent spec:** `docs/plans/2026-05-12-prgroom-cli-design.md` §3.2 / §3.3 / §5 / §8.3 / §8.8
> **Architecture:** `docs/architecture/prgroom/c4-l3-lifecycle.md` §"PR-memory: snapshot assembly + Decisions routing"

## Problem

The run-loop bead wired the §3.3 pipeline against a **no-op `_reply` skeleton** and a real `_resolve`. Two deterministic verb bodies are still hollow:

1. **`_reply`** must post real replies to reviewer items and route the fix agent's **CONTEXTUAL memory** to the PR (thread replies + a `## Decisions` block).
2. **`resolve-escalated`** must let a human flip one `escalated` item to a terminal disposition and release the lifecycle.

Both ship in **Phase 1**: a Phase-1 `run` that stopped before reply/resolve would regress today's full post-push flow.

A latent gap blocks (1): the fix path **validates** CONTEXTUAL memory entries (`audit_memory`) but never **surfaces** them — `MemoryAuditResult`/`FixRunResult` carry only `deferred` (non-CONTEXTUAL). Routing requires surfacing the valid CONTEXTUAL entries and carrying them to `_reply`.

This bead also **reorders the post-push pipeline** (see Locked decisions). The §3.5 hard-cap rework (gate at cycle entry, never run a fix that won't push) is **out of scope** — filed as `agents-config-abn9.8.18`.

## Locked decisions

- **Template-only reply bodies.** Each reply is rendered from the item's `Disposition` (kind / rationale / commits) via a canonical matrix. The fix agent's long-form `response_path` prose is **deferred** to a follow-up (parent spec §2: "schema deliberately omits a pre-rendered `reply_body` — rendered at reply verb time from current item state"; the bead AC does not require `response_path`).
- **Durable memory via a persisted `pending_memory` field.** `_fix` resolves CONTEXTUAL memory into `RoutedMemory` records and appends them to a new `PRGroomingState.pending_memory`; `_reply` routes them and clears the field. This is a **deliberate schema change** that reverses the architecture doc's "no schema change" summary line — the in-memory alternative loses the decision memo on *every* cycle that ends before `_reply` (the cap-trip break and any transient gh failure on push — both fall between `_fix`, which produces the memo, and `_reply`, which routes it), which is too lossy for a "the PR is the memory" design. The c4-l3-lifecycle doc is updated with this rationale. The field is omit-when-empty, so `schema_version` stays `1` (additive optional; old state files load with `pending_memory=[]`).
- **Idempotency is two distinct mechanisms.** Per-item replies dedup via the persisted `ReviewItem.replied` flag. Memory routing dedups with **no extra flag**: the `## Decisions` block is keyed by `(round, source-item)` (parse-existing → skip-existing → re-render wholesale → byte-identical re-run); thread-reply memory shares the same documented per-reply crash window as item replies (below).
- **Memory routing works wherever `pending_memory` is non-empty.** Because the field is durable, the standalone `prgroom reply` verb routes any memory a prior `fix`/`run` left pending — not just the in-`run` path.
- **Pipeline reorder:** `… → _push → _reply → _resolve → _rereview` (was `_push → _rereview → _reply → _resolve`). Reply/resolve close out the current round; rereview kicks off the next, so it goes **last** — and resolving the FIXED threads before re-requesting review hands the bot a clean slate. No data dependency required the old order; it was a carry-over from the legacy skill. (`run.py _build_pipeline` + §3.3 doc-sync.)
- **Resumable rereview trigger.** The reorder breaks the existing cycle-relative `_rereview` guard. Today `_rereview_guard = push_uploaded_commits_this_cycle(state, cycle_start_pushed_sha) AND has_required_reviewers_to_refresh(state)`. With rereview moved *after* the failure-prone `_reply`/`_resolve`, a cycle that pushes, persists, then dies in reply/resolve (PROPAGATE → cycle aborts) re-enters with `cycle_start_pushed_sha == last_pushed_head_sha` (the already-pushed SHA); push is a no-op, `push_uploaded_commits_this_cycle` is **false**, and rereview is skipped **forever** — Copilot is never re-poked. The fix replaces the cycle-relative half with a **persisted SHA comparison**: a new `last_rereviewed_sha` on state, guard = `push_awaiting_rereview(state) AND has_required_reviewers_to_refresh(state)` where `push_awaiting_rereview ≡ last_pushed_head_sha != last_rereviewed_sha`; `_rereview` stamps `last_rereviewed_sha = last_pushed_head_sha` on a successful re-request. The signal is durable, so the aborted cycle resumes correctly. `push_uploaded_commits_this_cycle` and `cycle_start_pushed_sha` **stay** — the end-of-cycle resolver's priority-4 `pushed_this_cycle` still reads them. (Rejected: **(A)** trigger on `has_required_reviewers_to_refresh` alone — per §3.4 it "reduces to *any required reviewers configured*", so it fires rereview on a no-push cycle whenever a required reviewer is `declined`, flipping `declined → requested` and **un-satisfying** G_REVIEWERS (`declined ∈ _REVIEWER_DONE`, `requested ∉`), blocking quiescence for commits that never changed — a semantic regression. **(B)** a `rereview_pending` bool — works, but the guard's reviewer conjunct gates the step, so in the reviewers-unpopulated MVP the flag dangles `True` indefinitely; the SHA reads cleaner and is operator-debuggable.) This is run-loop (§3.3) surgery folded into this bead.

## Component 1 — surface + resolve CONTEXTUAL memory in the fix path

The fix agent computes; the lifecycle applies. Surfacing rides the existing compute→apply seam.

- **`agent/memory_audit.py`** — `MemoryAuditResult` gains `routable: list[MemoryEntry]`. `audit_memory` appends a **valid** CONTEXTUAL entry (passes classification + exactly-one-of-content|path + known `target_hint`) to `routable`. Invalid entries still become `violations`; non-CONTEXTUAL still become `deferred`.
  - **Security (containment for `path`-form):** a CONTEXTUAL entry with `path` set is containment-checked the same way `memory_writes` is — anchored under `memory_dir`, lexical `normpath` + prefix test. An escape (`/etc/passwd`, `..` traversal) is a **hard `Severity.BLOCK` violation** (cluster-flipping), never routable, never read. Reading an unbounded `entry.path` and publishing it to the PR would be an exfiltration hole; today the audit only checks `memory_writes`, so this is a new, required check.
- **`agent/fix.py`** — `FixRunResult` gains `contextual_memory: list[MemoryEntry]`; `run_fix` threads `memory.routable` onto it. Pure; no I/O. **`target_hint` validation stays cluster-scoped** (no widening). `run_fix` is called with no `known_thread_ids`, so `_resolve_thread_ids` derives the known set from the **cluster's own items** (`{it.identity.thread_id for it in req.items}`); `SnapshotPaths` exposes only file paths, not the full snapshot's thread-id set, so there is no data path to validate against "every snapshot thread". This **fails safe**: a `target_hint` naming a thread outside the cluster is rejected as an unknown-hint audit failure (not routed), never published. Full-snapshot widening is deferred (a hint to a *real but non-cluster* thread would need the snapshot's thread-id set surfaced — out of scope here).
- **`lifecycle/fix.py`** — `_fix_one_cluster` resolves each routable entry into a `RoutedMemory` **while the agent's scratch files still exist** (inside the `fix()` `TemporaryDirectory`) and appends to `state.pending_memory`:
  - content-form → content used verbatim (sanitized for the Decisions line — see Component 3);
  - path-form → file read verbatim (AC: "path-form uses file contents verbatim"). **Two-layer containment.** The audit's lexical check (`agent/memory_audit.py`, `os.path.normpath` + prefix) is anchored under `memory_dir` but, by design, never calls `realpath` — so a **symlink inside `memory_dir` pointing at `/etc/passwd`** passes lexically and would be read-and-published. So `_fix_one_cluster` adds a **realpath/no-symlink check immediately before the read**: resolve the entry path with `os.path.realpath` and verify the resolved target is still under `realpath(memory_dir)`; an escape is a **hard `Severity.BLOCK` violation** (cluster-flipped, entry skipped, never read), matching the lexical layer's severity. A read failure on a contained, non-symlinked path is a soft `warn` (bookkeeping drift), entry skipped.

```python
@dataclass(frozen=True, slots=True)
class RoutedMemory:
    content: str            # resolved verbatim (inline content, or file body for path-form)
    round: int              # state.round at fix time (the round the decision was made)
    source_item: str        # "<cluster_id>#<ordinal>" — the (round, source-item) dedup key
    decided_by: str         # the fix agent id (e.g. "claude -p opus[1m]")
    target_hint: str | None = None   # thread node-id (PRRT_*) for a thread reply; None → Decisions block
    # to_dict / from_dict: omit target_hint when None (matches §2 serialization convention)
```

`source_item = f"{cluster_id}#{ordinal}"` (`ordinal` = the entry's index within that cluster's routable list) so one cluster emitting two thread-less decisions in a round yields two distinct keys (the §8.3 worked example does this), while the same entry re-resolved yields the same key. `fix_pr` mutates `state.pending_memory` on the working copy it already returns — no out-param; the caller's single `store.write` persists memory atomically with the dispositions it came from.

## Component 2 — `run.py` (reorder + reply wiring + resumable rereview guard)

- **Reorder** `_build_pipeline`: `cluster → fix → cap-guard → push → reply → resolve → rereview`. (Cap-guard stays where it is — moving it is `abn9.8.18`.)
- **Rereview guard (§6 resumability fix).** `_rereview_guard` becomes `push_awaiting_rereview(ctx.state) AND has_required_reviewers_to_refresh(ctx.state)` — dropping the cycle-relative `push_uploaded_commits_this_cycle(state, cycle_start_pushed_sha=…)` half for a persisted SHA comparison (`push_awaiting_rereview`, new pure predicate in `lifecycle/predicates.py`, ≡ `state.last_pushed_head_sha != state.last_rereviewed_sha`). `cycle_start_pushed_sha` is still snapshotted at cycle entry and still feeds the resolver's `pushed_this_cycle` (priority-4) — only the rereview guard changes. See Locked decisions for the rejected alternatives.
- The `reply` ctx-adapter passes `gh=ctx.gh, ref=ctx.ref` (it already has state). No `RunContext` memory field — memory lives in `state.pending_memory`, so there is nothing to thread or reset per cycle.

> **`lifecycle/rereview.py` (run-loop §3.3 surgery).** `rereview_pr`, after re-requesting all stale required reviewers, stamps `state.last_rereviewed_sha = state.last_pushed_head_sha` (on the deepcopy it returns; the caller persists). A POST failure mid-dance propagates and discards the copy, leaving `last_rereviewed_sha` un-advanced — so the next run re-attempts (preserving the existing crash-safety). The stamp is what makes `push_awaiting_rereview` go false after a clean rereview; combined with the reviewer-status flip to `requested` (which empties the `has_required_reviewers_to_refresh` set), a second rereview in the same cycle is doubly a no-op.

## Component 3 — `_reply` body (`lifecycle/reply.py`)

```python
def reply_pr(state: PRGroomingState, *, gh: GhClient, ref: PRRef,
             warn: Callable[[str], None] = default_warn) -> PRGroomingState:
```

Works on a deepcopy; caller persists once (verb-atomic). No phase change, no `last_error` (§3.2 reply row).

### Per-item replies (PR-public text, no internal jargon)

For each item with `replied is False` and a *replyable* disposition, render the body, POST it, set `replied = True`.

| `disposition.kind` | Body |
|---|---|
| `fixed` | `Fixed in <sha>. <rationale>` (first commit SHA; rationale omitted if empty) |
| `already_addressed` | `Already addressed in <sha>.` |
| `skipped` / `deferred` / `wont_fix` | `<rationale>` (user-facing) |
| `escalated` | `Captured for follow-up; will respond on a later push to this PR or in a related issue.` — cap-exceeded variant (`Round limit reached on this PR; deferring further iterations to a human reviewer.`) when `rationale` names the cap |
| `failed` | **no reply** (internal failure, pre-human-gate) |

`escalated` is replied **regardless of `escalation_filed`** — that flag is set later, by the terminal flush, which runs *after* `_reply` in the pipeline, so gating on it would never fire in-cycle.

Endpoint by `ItemKind` (gh REST via `gh.rest`):

| `kind` | Endpoint |
|---|---|
| `review_thread` | `POST repos/{o}/{r}/pulls/{n}/comments/{reply_id}/replies` `-f body=` where `reply_id = identity.reply_to_comment_id or int(identity.gh_id)` |
| `issue_comment` / `review_summary` | `POST repos/{o}/{r}/issues/{n}/comments` `-f body=` |

`reply_to_comment_id` is `0` for a **top-level** inline review comment (poll sets it only on nested replies); the comment's own id is `gh_id`. Replying to `…/comments/{reply_to_comment_id or gh_id}/replies` posts into that thread for both top-level and nested cases. No-op exit when every item is already replied.

### CONTEXTUAL memory routing (drain `state.pending_memory`)

For each `RoutedMemory rm` in `state.pending_memory`:

- **`rm.target_hint` set → thread reply.** GraphQL `addPullRequestReviewThreadReply(input:{pullRequestReviewThreadId: rm.target_hint, body: rm.content})` — node-id native (mirrors `resolve_pr`).
- **`rm.target_hint` is `None` → `## Decisions` block.** `GET …/pulls/{n}` → `body`, merge, `PATCH …/pulls/{n}` `-f body=`. **Not** a git commit, **not** a phase change.

After routing all entries, clear `state.pending_memory = []`.

**Decisions block (sentinels reused from `snapshot.py`):**

```markdown
<!-- prgroom:decisions:start -->
## Decisions
<!-- d:r1:c-abc#0 --> - **[r1]** <sanitized content> _(decided_by: <decided_by>)_
<!-- prgroom:decisions:end -->
```

**Merge (`merge_decisions_block(body, entries) -> body`), pure:** parse existing lines into ordered `(round, source_item) → line` via the leading machine marker `<!-- d:r<round>:<source_item> -->` (exact, never markdown-prose guessing); for each new entry skip if its key exists (never overwrite, never delete prior) else append; re-render wholesale and splice between sentinels (creating block + heading if absent). Deterministic key→line rendering ⇒ **byte-identical** same-round re-run; an existing key for item A does not block a new same-round entry for item B. **Content is sanitized** before rendering — newlines collapsed to spaces and any `<!-- … -->` sequences stripped — so a one-line entry can't corrupt the marker-based parser.

**Crash-window note (documented, mirrors `resolve_pr`).** `_reply` is verb-atomic (deepcopy → caller persists once). A failure *mid-verb* discards the verb's `replied` marks and re-runs: per-item replies and thread-reply memory may double-post (gh comment POSTs are not server-idempotent, unlike `resolveReviewThread`); the Decisions block is fully idempotent (skip-existing-key). Persistence guarantees no memory is *lost* (it stays in `pending_memory` until a clean reply clears it); it does not eliminate the non-idempotent-POST double-post window. The legacy skill carried the same per-reply window.

## Component 4 — `resolve-escalated`

```python
def resolve_escalated_pr(state: PRGroomingState, *, item_id: str,
    as_disposition: DispositionKind, rationale: str, commits: list[str],
    decided_by: str, now: datetime) -> PRGroomingState:   # decided_by = "human:<git-user>"
```

Lock-held internal (own lock via the CLI `with_lock` wrapper). Behavior (§3.2 `resolve-escalated` row, §5 Contract C):

1. `as_disposition ∈ {fixed, skipped, deferred, wont_fix}` is enforced **at the CLI boundary** by a typer `enum` choice (a `StrEnum` subset of `DispositionKind` exposing exactly those four), so an invalid `--as` is rejected by typer/click with exit 2 **before** `resolve_escalated_pr` runs — *not* via a raised `PrgroomError` (`PRECONDITION_USER_ERROR` is a `Tier`, not an `ErrorCode`, so it cannot be raised). The lifecycle function still type-restricts its `as_disposition` parameter. For `fixed`, `commits` must be non-empty (the reply matrix renders `Fixed in <sha>`); empty → `PRECONDITION_FIXED_NEEDS_COMMITS` (new code, USER_ERROR tier, exit 2).
2. If no item in `state.items` is `escalated` → `PRECONDITION_NO_ESCALATIONS` (existing no-work code, exit 0).
3. Find the item by `identity.gh_id == item_id` whose disposition is `escalated`; absent or not-escalated → `PRECONDITION_ITEM_NOT_ESCALATED` (new code, USER_ERROR tier, exit 2).
4. Replace its `disposition` with `Disposition(kind=as_disposition, decided_at=now, decided_by=decided_by, rationale=rationale, commits=commits)`.
5. **Phase advance** — only from `human-gated`: → `fixes-pending` **iff** no `escalated` remain **and** no `failed` item **and** `state.last_error ∉ BlockingErrorCodes`; else stay. From `fixes-pending`, phase unchanged.
6. **Never** clears cap / state-corrupt / failed gating; **never** increments `round`. Re-runnable.

`BlockingErrorCodes` reuses the existing `errors.BlockingErrorCodes` frozenset, which is a **superset** of §3.2's example list — it also includes `RUNTIME_GIT_TERMINAL` (a deliberate §3.6 extension, per the constant's own comment). Reuse keeps the one source of truth; the §3.2 list is illustrative, not exhaustive.

**CLI (`cli.py`)** — replace the `resolve-escalated` skeleton with `read → resolve_escalated_pr → write` under `with_lock`. The skeleton's `as_: str` option is retyped to the four-value `StrEnum` (typer renders it as a choice; invalid → exit 2 at parse), a new `--commits a,b,c` option is parsed into a list (split on `,`, drop empties), and `decided_by = "human:" + _git_user()` is derived. `_git_user()` reads `git config user.name` (fallback `user.email`) via a new `GitClient.config_user()` adapter method (subprocess stays behind the adapter boundary). Replace the standalone `reply` skeleton with `read → reply_pr → write` (`PRECONDITION_NO_ITEMS` when no items; no-op exit 0 when nothing unreplied and `pending_memory` empty; `merged` → no-op).

## Component 5 — reply rendering via the existing engine (OBS-10)

Reply bodies render through the **existing** `agent/prompt_loader.PromptTemplate` (strict `format_map`, loud on a missing key) — the same engine agent prompts already use, satisfying OBS-10's "load through the same engine" **without** a second engine and **without** relocating the module (keeps this bead's footprint tight). The per-disposition bodies are a small Python matrix of `PromptTemplate`s in `lifecycle/reply.py` (the single owner of reply text, mirroring legacy `render-reply-bodies.sh`), not seven `.tmpl` files. A future bead may lift `PromptTemplate` into a neutral module if a third consumer appears.

## Schema (`prsession/state.py`)

- `PRGroomingState` gains `pending_memory: list[RoutedMemory] = field(default_factory=list)`, omit-when-empty in `to_dict`, `.get("pending_memory", [])` in `from_dict`. `schema_version` stays `1`.
- `PRGroomingState` also gains `last_rereviewed_sha: str = ""` (the §6 resumable-rereview marker), omit-when-empty in `to_dict`, `.get("last_rereviewed_sha", "")` in `from_dict` — same additive-optional pattern as the sibling `last_pushed_head_sha` / `last_poll_sha`. `schema_version` stays `1` (old state files load with `""`).
- `RoutedMemory` gets `to_dict`/`from_dict` (omit `target_hint` when `None`). It lives in `prsession/state.py` (it is persisted state), imported by `lifecycle/fix.py` and `lifecycle/reply.py`.
- Written atomically with fix's dispositions and cleared atomically by reply (the same per-verb `store.write`), so `pending_memory` and the dispositions it derives from never diverge.

## Error / precondition codes

Already registered: `PRECONDITION_NO_ITEMS`, `PRECONDITION_NO_UNREPLIED`, `PRECONDITION_NO_ESCALATIONS`, `PRECONDITION_NO_STATE`; gh failures map through the existing adapter tiers (`RUNTIME_GH_TRANSIENT` / `RUNTIME_GH_TERMINAL` / `RUNTIME_GRAPHQL_FAILED`). `errors.BlockingErrorCodes` reused.

**Two new codes in `errors.py`** (the round-2 fix — `PRECONDITION_USER_ERROR` is a `Tier`, *not* an `ErrorCode`, so the earlier draft's "raise `PRECONDITION_USER_ERROR`" was unbuildable):

- `PRECONDITION_FIXED_NEEDS_COMMITS` — `resolve-escalated --as fixed` with empty `--commits`.
- `PRECONDITION_ITEM_NOT_ESCALATED` — the named `item_id` is absent or its disposition is not `escalated` (and ≥1 escalated item exists).

Both are added to the `ErrorCode` enum with `_REGISTRY` what/why/how entries, and **deliberately NOT** added to `_NO_WORK_CODES` — so `ErrorCode.precondition_tier()` returns `Tier.PRECONDITION_USER_ERROR` (exit 2), the correct "operator passed a bad argument" tier. (Tier assignment is by the explicit `_NO_WORK_CODES` allow-list, not by name prefix.)

## Testing (fakes-not-mocks, `mypy --strict`, error-path coverage)

`FakeGh` recording `rest`/`graphql` calls; assert exact method + path + body/variables.

- **Per-item matrix:** one case per disposition→body row; each `kind`→endpoint; `review_thread` top-level (`reply_to_comment_id == 0`) replies to `gh_id`; nested reply replies to the parent; `failed` gets no reply; `escalated` replies regardless of `escalation_filed`.
- **Idempotency:** `replied=True` per posted item; no-op when all replied; standalone `PRECONDITION_NO_UNREPLIED` only under `--no-prework`-equivalent.
- **Memory routing:** `target_hint` → exact GraphQL mutation + variables; thread-less → GET-merge-PATCH; **fit-test:** byte-identical block on same-round re-run **and** skip-existing-key while appending a different same-round item; content with newlines / `<!-- -->` sanitized; `pending_memory` cleared after a clean route.
- **Durability:** a cap-trip / transient-failure cycle leaves `pending_memory` persisted; a later `_reply` (after re-arm) routes it (not lost).
- **Surfacing + security:** `audit_memory` routable for a valid CONTEXTUAL entry; lexical `path`-form escape (`/etc/passwd`, `..`) → BLOCK violation, cluster-flipped, never read; **realpath layer** — a symlink *inside* `memory_dir` resolving outside it → BLOCK at the `_fix_one_cluster` read-time check, never read/published (the lexical-pass-but-realpath-fail case); `run_fix` carries `contextual_memory`; `fix_pr` resolves content-form + audited path-form into `pending_memory`; `target_hint` validated against the **cluster's** thread ids (a hint to a non-cluster thread → unknown-hint audit failure, not routed).
- **§6 resumable rereview:** `push_awaiting_rereview` true after a push, false after `rereview_pr` stamps `last_rereviewed_sha`; **resumability** — push succeeds, `_reply`/`_resolve` PROPAGATE (cycle aborts), next run's no-op push leaves `push_awaiting_rereview` true so rereview still fires (vs. the old cycle-relative guard, which skipped it); **no spurious rereview** — a no-push cycle with a `declined` required reviewer does NOT fire rereview (guard's `push_awaiting_rereview` half is false), so quiescence is not blocked; `rereview_pr` POST failure leaves `last_rereviewed_sha` un-advanced (re-attempt next run); `last_rereviewed_sha` round-trips.
- **resolve-escalated:** flip per valid `--as`; invalid `--as` rejected at the typer boundary (exit 2, no lifecycle call); `fixed` with empty `commits` → `PRECONDITION_FIXED_NEEDS_COMMITS` (exit 2); no escalated items → `PRECONDITION_NO_ESCALATIONS` (exit 0); wrong/absent item with escalations present → `PRECONDITION_ITEM_NOT_ESCALATED` (exit 2); phase advance gated by each of {escalated-remain, failed-present, each `BlockingErrorCode`}; no `round` increment; re-run clean.
- **Schema:** `pending_memory`/`RoutedMemory` round-trip; old state file (no field) loads with `[]`.
- **CLI:** `reply` + `resolve-escalated` wiring (CliRunner), `--commits` parse, `_git_user` derivation, exit codes via `handle_cli_error`.
- **e2e:** a `run` cycle fixes an item, pushes, replies, routes one thread-reply + one Decisions entry, resolves the thread, then rereviews — full-fidelity reply/resolve.

## Review feedback

**Round 1 — Codex `gpt-5.5` adversarial review (NO-GO → resolved):** in-memory threading lost memory on cap-trip/transient-failure → **persist `pending_memory`** (Locked decisions, Schema). `reply_to_comment_id == 0` for top-level comments → **`reply_to_comment_id or gh_id`** (Component 3). Unchecked `MemoryEntry.path` → **hard containment check** (Component 1). `BlockingErrorCodes` is a §3.2 superset → reuse + documented (Component 4). `escalation_filed` set after `_reply` → **reply to escalated unconditionally** (Component 3). Plus: Decisions content sanitization, `--as fixed` requires commits, widened `target_hint` validation, template engine reused-in-place (no relocation), and the missed file changes (`config_user`, `__init__` export, fakes) — all folded in.

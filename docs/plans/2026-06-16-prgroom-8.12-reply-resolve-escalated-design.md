# Design: prgroom deterministic verbs — `_reply` (+ CONTEXTUAL memory routing) + `resolve-escalated`

> **Bead:** agents-config-abn9.8.12 · **Parent spec:** `docs/plans/2026-05-12-prgroom-cli-design.md` §3.2 / §3.3 / §5 / §8.3 / §8.8
> **Architecture:** `docs/architecture/prgroom/c4-l3-lifecycle.md` §"PR-memory: snapshot assembly + Decisions routing"

## Problem

The run-loop bead wired the §3.3 pipeline (`_poll → _cluster → _fix → [cap] → _push → [_rereview] → _reply → _resolve`) against a **no-op `_reply` skeleton** and a real `_resolve`. Two deterministic verb bodies are still hollow:

1. **`_reply`** must post real replies to reviewer items and route the fix agent's **CONTEXTUAL memory** to the PR (thread replies + a `## Decisions` block).
2. **`resolve-escalated`** must let a human flip one `escalated` item to a terminal disposition and release the lifecycle.

Both ship in **Phase 1** (not Phase 2): a Phase-1 `run` that stopped before reply/resolve would regress today's full post-push flow.

A latent gap blocks (1): the fix path **validates** CONTEXTUAL memory entries (`audit_memory`) but never **surfaces** them — `MemoryAuditResult`/`FixRunResult` carry only `deferred` (non-CONTEXTUAL). Routing requires surfacing the valid CONTEXTUAL entries and threading them to `_reply`.

## Locked decisions

- **Template-only reply bodies.** Each reply is rendered from the item's `Disposition` (kind / rationale / commits) via a canonical matrix. The fix agent's long-form `response_path` prose is **deferred** to a follow-up (the parent spec §2 "schema deliberately omits a pre-rendered `reply_body` — rendered at reply verb time from current item state"; the bead AC does not require `response_path`).
- **No schema change.** CONTEXTUAL memory is routed inside `_reply` from the fix output's memory channel, threaded **in-memory** through one `run` invocation — never persisted to `PRGroomingState` (architecture doc, "no new verb, no schema change"; the PR is the durable store).
- **Idempotency is two distinct mechanisms.** Per-item replies dedup via the persisted `ReviewItem.replied` flag. Memory routing dedups with **no persisted flag**: the `## Decisions` block is keyed by `(round, source-item)` against the PR body itself, and a crash-and-re-run finds `_fix` a no-op (items already dispositioned) → empty memory channel → nothing re-routes.
- **Memory routing is run-pipeline only.** The standalone `prgroom reply` verb posts per-item replies from state; it routes no memory (no fix output exists outside a `run`).

## Component 1 — surface CONTEXTUAL memory from the fix path

The fix agent computes; the lifecycle applies. Surfacing rides the existing compute→apply seam.

- **`agent/memory_audit.py`** — `MemoryAuditResult` gains `routable: list[MemoryEntry]`. `audit_memory` appends a **valid** CONTEXTUAL entry (passes classification + exactly-one-of-content|path + known `target_hint`) to `routable`. Invalid entries still become `violations`; non-CONTEXTUAL still become `deferred`. (A valid CONTEXTUAL entry was previously validated and dropped.)
- **`agent/fix.py`** — `FixRunResult` gains `contextual_memory: list[MemoryEntry]`; `run_fix` threads `memory.routable` onto it. Pure; no I/O.
- **`lifecycle/fix.py`** — `_fix_one_cluster` resolves each routable entry into a `RoutedMemory` **while the agent's scratch files still exist** (inside the `fix()` `TemporaryDirectory`), and appends to a caller-supplied sink:
  - content-form → content used verbatim;
  - path-form → file read verbatim (the bead AC: "path-form uses file contents verbatim"); a path read failure is a soft `warn` (bookkeeping drift, not a cluster breach — consistent with §8.6 declared-but-missing), the entry is skipped.

```python
@dataclass(frozen=True, slots=True)
class RoutedMemory:
    content: str            # resolved verbatim (inline content, or file body for path-form)
    round: int              # state.round at fix time (the round the decision was made)
    source_item: str        # "<cluster_id>#<ordinal>" — the (round, source-item) dedup key
    decided_by: str         # the fix agent id (e.g. "claude -p opus[1m]")
    target_hint: str | None = None   # thread node-id (PRRT_*) for a thread reply; None → Decisions block
```

`source_item` is assigned at resolution time as `f"{cluster_id}#{ordinal}"`, where `ordinal` is the entry's index within that cluster's routable list — so a single cluster emitting two thread-less decisions in one round yields two distinct keys (the §8.3 worked example does exactly this), while the same entry re-resolved yields the same key.

`fix_pr` gains an optional `routed_memory_out: list[RoutedMemory] | None = None` it appends to (mutable sink, mirroring how `sink.emit` carries escalations out). When `None` (the standalone `fix` verb), routing data is simply not collected.

## Component 2 — thread the channel through `run.py`

- `RunContext` gains `pending_memory: list[RoutedMemory] = field(default_factory=list)` (in-memory, not part of the persisted schema).
- The `fix` ctx-adapter resets `ctx.pending_memory = []` then passes it as `routed_memory_out` (fresh per cycle — fix runs once per cycle, before reply).
- The `reply` ctx-adapter passes `gh=ctx.gh, ref=ctx.ref, routed_memory=ctx.pending_memory`.

Per-cycle reset + fix-before-reply ordering means `pending_memory` holds exactly this cycle's routable entries when `_reply` runs.

## Component 3 — `_reply` body (`lifecycle/reply.py`)

```python
def reply_pr(
    state: PRGroomingState,
    *,
    gh: GhClient,
    ref: PRRef,
    routed_memory: Sequence[RoutedMemory] = (),
    warn: Callable[[str], None] = default_warn,
) -> PRGroomingState:
```

Works on a deepcopy; caller persists. No phase change, no `last_error` (§3.2 reply row).

### Per-item replies (PR-public text, no internal jargon)

For each item with `replied is False` and a *replyable* disposition, render the body, POST it, set `replied = True`.

| `disposition.kind` | Body |
|---|---|
| `fixed` | `Fixed in <sha>. <rationale>` (first commit SHA; rationale omitted if empty) |
| `already_addressed` | `Already addressed in <sha>.` |
| `skipped` / `deferred` / `wont_fix` | `<rationale>` (user-facing) |
| `escalated` **and** `escalation_filed` | `Captured for follow-up; will respond on a later push to this PR or in a related issue.` — cap-exceeded variant (`Round limit reached on this PR; deferring further iterations to a human reviewer.`) when `rationale` names the cap |
| `failed` | **no reply** (internal failure, pre-human-gate) |
| `escalated` **and not** `escalation_filed` | **no reply** (not yet surfaced) |

Endpoint by `ItemKind` (gh REST via `gh.rest`):

| `kind` | Endpoint |
|---|---|
| `review_thread` | `POST repos/{o}/{r}/pulls/{n}/comments/{reply_to_comment_id}/replies` `-f body=` |
| `issue_comment` / `review_summary` | `POST repos/{o}/{r}/issues/{n}/comments` `-f body=` |

A `review_thread` item with `reply_to_comment_id == 0` is degraded (poll could not map it): skip with a `warn`, leave `replied = False` (mirrors `resolve_pr`'s degraded-thread handling). No-op exit when every item is already replied.

### CONTEXTUAL memory routing

For each `RoutedMemory rm`:

- **`rm.target_hint` set → thread reply.** GraphQL `addPullRequestReviewThreadReply(input:{pullRequestReviewThreadId: rm.target_hint, body: rm.content})` — node-id native, mirroring `resolve_pr`'s GraphQL use. (`target_hint` was audited to name a real thread before becoming routable.)
- **`rm.target_hint` is `None` → `## Decisions` block.** Read PR body (`GET repos/{o}/{r}/pulls/{n}` → `body`), merge, `PATCH repos/{o}/{r}/pulls/{n}` `-f body=`. **Not** a git commit, **not** a phase change.

**Decisions block (sentinels owned by `snapshot.py`, reused):**

```markdown
<!-- prgroom:decisions:start -->
## Decisions
- **[r1] <rm.content>** _(decided_by: <rm.decided_by>)_
<!-- prgroom:decisions:end -->
```

**Merge algorithm (`merge_decisions_block(body, new_entries) -> body`), pure:**

1. Parse the existing block (between sentinels) into ordered `(round, source_item) → line`. The key is embedded per line as a leading machine marker `<!-- d:r<round>:<source_item> -->` so parsing is exact, never a markdown-prose guess; the human-visible `- **[r<round>] …**` follows.
2. For each new entry, in order: if its `(round, source_item)` key is already present, **skip** (never overwrite, never delete prior); else append.
3. Re-render the whole block wholesale from the ordered key→line map and splice between sentinels (creating the block + `## Decisions` heading if absent).

Re-rendering wholesale from a deterministic key→line map yields a **byte-identical** block on a same-round re-run, and an existing key for item A does not block appending a new same-round entry for item B. This is the AC's required fit-test.

**Crash-window note (documented limitation, mirrors `resolve_pr`).** A crash *between* a gh POST and the end-of-verb state write re-posts that item's reply on the next run — gh comment POSTs are not server-idempotent (unlike `resolveReviewThread`). The AC's "never double-posts" guarantee covers the *memory* path, which is crash-safe via the empty-channel-on-rerun property. The legacy `reply-and-resolve-pr-threads` skill carried the same per-reply window.

## Component 4 — `resolve-escalated`

```python
def resolve_escalated_pr(
    state: PRGroomingState,
    *,
    item_id: str,
    as_disposition: DispositionKind,
    rationale: str,
    commits: list[str],
    decided_by: str,            # "human:<git-user>", derived by the CLI
    now: datetime,
) -> PRGroomingState:
```

Lock-held internal (own lock via the CLI `with_lock` wrapper). Behavior (§3.2 `resolve-escalated` row, §5 Contract C):

1. Validate `as_disposition ∈ {fixed, skipped, deferred, wont_fix}` (else `PRECONDITION_USER_ERROR`).
2. If no item in `state.items` is `escalated` → `PRECONDITION_NO_ESCALATIONS`.
3. Find the item by `identity.gh_id == item_id` whose disposition is `escalated`; if the named item is absent or not escalated → `PRECONDITION_USER_ERROR` (named the wrong item).
4. Replace its `disposition` with `Disposition(kind=as_disposition, decided_at=now, decided_by=decided_by, rationale=rationale, commits=commits)`.
5. **Phase advance** — only from `human-gated`: → `fixes-pending` **iff** no `escalated` remain **and** no `failed` item **and** `state.last_error ∉ BlockingErrorCodes` (reuse the existing `errors.BlockingErrorCodes` frozenset); else stay `human-gated`. From `fixes-pending` the phase is unchanged.
6. **Never** clears cap / state-corrupt / failed gating; **never** increments `round`. Re-runnable (a second run with the now-non-escalated item hits step 2/3 cleanly).

**CLI (`cli.py`)** — replace the `resolve-escalated` skeleton with `read → resolve_escalated_pr → write` under `with_lock`, parsing `--commits a,b,c` into a list and deriving `decided_by = "human:" + _git_user()`. `_git_user()` reads `git config user.name` (falling back to `user.email`) via a new `GitClient.config_user()` adapter method, keeping subprocess access behind the adapter boundary. Also replace the standalone `reply` skeleton with `read → reply_pr → write` (per-item replies only; `PRECONDITION_NO_ITEMS` when state has no items; no-op exit 0 when none unreplied; `merged` → no-op).

## Component 5 — shared template engine (OBS-10 unify)

The reply renderer is the "shared reply-rendering engine" the OBS-10 note anticipated. Promote `agent/prompt_loader.py`'s `PromptTemplate` + `_StrictMapping` (strict `format_map`, loud on a missing key) into a neutral `prgroom/templates.py`; `load_prompt` (agent prompts) and the reply matrix both render through it. The per-disposition reply bodies are a small Python matrix of `PromptTemplate`s (mirroring legacy `render-reply-bodies.sh` as the single owner of reply text) rendered through the shared engine — not seven `.tmpl` files. `agent/prompt_loader.py` re-exports for its existing importers; no behavior change to cluster/fix prompt loading.

## Error / precondition codes (all already registered)

`PRECONDITION_NO_ITEMS`, `PRECONDITION_NO_UNREPLIED`, `PRECONDITION_NO_ESCALATIONS`, `PRECONDITION_USER_ERROR`, `PRECONDITION_NO_STATE`; gh failures map through the existing adapter tiers (`RUNTIME_GH_TRANSIENT` / `RUNTIME_GH_TERMINAL` / `RUNTIME_GRAPHQL_FAILED`). `errors.BlockingErrorCodes` reused verbatim.

## Testing (fakes-not-mocks, `mypy --strict`, error-path coverage)

`FakeGh` recording `rest`/`graphql` calls (extend the existing fake); assert exact method + path + body.

- **Per-item matrix:** one case per disposition→body row; each `kind`→endpoint; degraded `review_thread` (id 0) skipped + warned; `failed`/unfiled-`escalated` get no reply.
- **Idempotency:** `replied=True` set per posted item; no-op when all replied; `PRECONDITION_NO_UNREPLIED` only under direct `--no-prework`-equivalent path.
- **Memory routing:** `target_hint` → exact GraphQL mutation + variables; thread-less → GET-merge-PATCH; **fit-test:** byte-identical block on same-round re-run **and** skip-existing-key while appending a different same-round item.
- **Surfacing:** `audit_memory` routable for a valid CONTEXTUAL entry; `run_fix` carries `contextual_memory`; `fix_pr` resolves content-form + path-form into `RoutedMemory` and appends to the sink; path-read failure → soft warn, entry skipped.
- **resolve-escalated:** flip for each valid `--as`; `PRECONDITION_NO_ESCALATIONS`; wrong/absent item → `PRECONDITION_USER_ERROR`; phase advance gated by each of {escalated-remain, failed-present, each `BlockingErrorCode`}; no `round` increment; re-run clean.
- **CLI:** `reply` + `resolve-escalated` wiring (CliRunner), `--commits` parse, `_git_user` derivation, exit codes via `handle_cli_error`.
- **e2e:** this bead carries the full-fidelity reply/resolve end-to-end — a `run` cycle that fixes an item, replies, routes one thread-reply + one Decisions entry, and resolves the thread.

## Out of scope

- The fix agent's `response_path` long-form reply text (deferred follow-up).
- The 5-class repo-wide router for non-CONTEXTUAL memory (`agents-config-abn9.23.4`) — non-CONTEXTUAL stays accepted-but-deferred (logged), preserving the channel shape so the router extends without a contract change.
- A `memory-route` verb, ADR-writing, `## Decisions` compaction/pruning, cross-PR memory (parent spec §8.8).
- The §3.3 pipeline shell, `_resolve`, `monitor-pr` skill (other beads).
